\chapter{Projeto do Sistema}
\label{capitulo:projeto}

Este capítulo apresenta os principais elementos de projeto do simulador \textit{Royal Legacy}. São detalhadas a arquitetura do sistema, os protótipos das interfaces, a modelagem de dados para persistência, os diagramas estruturais e comportamentais baseados na Unified Modeling Language (UML), bem como a representação lógica dos algoritmos centrais de validação e comunicação.

As decisões de projeto aqui expostas visam atender aos requisitos de modularidade, customização visual e execução assíncrona da Inteligência Artificial, conforme justificado na seção \ref{secao:justificativa}.

\section{Arquitetura do projeto}

A arquitetura do \textit{Royal Legacy} foi projetada seguindo o princípio da separação de responsabilidades, dividindo o sistema em camadas distintas que se comunicam através de protocolos padronizados. Essa abordagem é fundamental para garantir que o processamento pesado da IA não impacte a fluidez da interface gráfica \cite{Fowler2003}.

A Figura \ref{figura:arquitetura_sistema} apresenta o diagrama de implantação que ilustra como as partes do sistema foram divididas e como elas se comunicam.

%\figura{caminho}{tamanho cm}{legenda}{referencia}
\figura{./imagens/diagrama_1}{12cm}{Diagrama de Implantação da arquitetura do Royal Legacy.}{figura:arquitetura_sistema}

O fluxo de comunicação segue o modelo Cliente-Servidor local:
\begin{enumerate}
	\item \textbf{Camada de Apresentação (Godot Engine):} Atua como o cliente (Frontend). Gerencia a interface, captura entradas do usuário e envia o estado do jogo (formato FEN) para o intermediário.
	\item \textbf{Camada de Comunicação (\textit{Middleware} Python):} Um \textit{script} que roda em segundo plano, agindo como ponte. Ele orquestra a execução da IA e traduz os comandos.
	\item \textbf{Camada Lógica (Stockfish Binary):} O motor de xadrez (Backend). Recebe configurações via protocolo UCI e devolve os cálculos matemáticos.
\end{enumerate}

\section{Diagrama Entidade Relacionamento (DER)}

Dado que o \textit{Royal Legacy} é um simulador local focado na jogabilidade e não na gestão complexa de usuários, optou-se por não utilizar um Banco de Dados Relacional tradicional. A persistência de dados é realizada através do objeto \textit{Singleton} \texttt{Global.gd} e arquivos de configuração locais, simplificando a arquitetura \cite{Date2003}.

A Figura \ref{figura:der_sistema} apresenta o Diagrama Entidade Relacionamento que rege a organização lógica das preferências do sistema.

\figura{./imagens/diagrama_2.png}{12cm}{Diagrama Entidade Relacionamento para persistência de preferências locais.}{figura:der_sistema}

O DER segue as normas formais de modelagem:
\begin{itemize}
	\item \textbf{Configuracao\_Local:} Centraliza as preferências do usuário. Armazena de forma independente os níveis de volume para o canal mestre (\textit{Master}), trilha sonora (\textit{Musica}) e efeitos sonoros (\textit{Efeitos}), além da referência ao tema visual ativo.
	\item \textbf{Tema\_Visual:} Estrutura os códigos de cores (casas claras e escuras) e identificadores de \textit{assets} para a customização dinâmica da interface do tabuleiro.
\end{itemize}

Como o simulador prioriza sessões de jogo independentes, o modelo não contempla uma entidade de histórico de partidas, focando-se na persistência da experiência de uso e customização estética. Todos os campos modelados possuem representação direta nos \textit{scripts} de interface \texttt{tela\_sons.gd} e \texttt{tela\_temas.gd}.

\section{Diagrama de Casos de Uso}

O Diagrama de Casos de Uso (Figura \ref{figura:caso_uso}) descreve as funcionalidades oferecidas pelo simulador e as interações entre o usuário e o motor de IA.

\figura{./imagens/diagrama_3.png}{12cm}{Diagrama de Casos de Uso do simulador.}{figura:caso_uso}

Os principais casos de uso identificados são:
\begin{itemize}
	\item \textbf{Configurar Preferências:} O usuário interage com as interfaces \texttt{tela\_sons.gd} e \texttt{tela\_temas.gd} para ajustar volumes e a estética do jogo.
	\item \textbf{Iniciar Partida:} Gerenciado pelo \texttt{estruturador.gd}, permite definir o modo de jogo e o nível de dificuldade.
	\item \textbf{Movimentar Peças:} Ação principal de interação no tabuleiro (\texttt{base\_xadrez.gd}), onde o sistema valida as regras do xadrez.
	\item \textbf{Calcular Jogada:} Caso de uso executado pelo ator secundário (Stockfish) via \texttt{chess\_bridge.py}, disparado de forma assíncrona quando o modo PvE está ativo.
\end{itemize}

\section{Diagrama de Classes}

A organização estrutural do código na Godot Engine e a integração com o \textit{middleware} Python podem ser representadas formalmente através de um Diagrama de Classes UML. A Figura \ref{figura:diagrama_classes} apresenta as principais classes de domínio do \textit{Royal Legacy}, seus atributos, métodos e relacionamentos.

\figura{./imagens/diagrama_4.png}{14cm}{Diagrama de Classes detalhando a estrutura em GDScript e Python.}{figura:diagrama_classes}

Destacam-se as seguintes classes no funcionamento do sistema:
\begin{itemize}
	\item \textbf{Global (Singleton):} Classe \textit{Autoload} que persiste o estado da aplicação entre as trocas de cena, armazenando variáveis como \texttt{tema\_escolhido} e \texttt{dificuldade\_escolhida}.
	\item \textbf{Estruturador:} Atua como o nó raiz (\texttt{estruturador.gd}), contendo um \textit{Array} de nós para instanciar e gerenciar a transição entre as interfaces gráficas através do método \texttt{mostrar\_tela()}.
	\item \textbf{BaseXadrez:} Classe central do jogo (\texttt{base\_xadrez.gd}). Detém a matriz do tabuleiro, executa a validação de regras e possui o método \texttt{matriz\_para\_fen()}, essencial para traduzir o estado do jogo. Também gerencia a \texttt{ia\_thread}.
	\item \textbf{GameManager:} Classe (\texttt{game\_manager.gd}) responsável por monitorar o fluxo da partida, emitindo sinais de xeque e definindo o encerramento do jogo via \texttt{end\_game()}.
	\item \textbf{ChessBridge:} \textit{Script} Python (\texttt{chess\_bridge.py}) independente, que recebe a string FEN e a dificuldade, instanciando o motor Stockfish para calcular e retornar a jogada.
\end{itemize}

\section{Algoritmos Implementados}

Nesta seção, detalha-se a lógica de conversão do estado do jogo e a comunicação com a IA. O Algoritmo \ref{alg:matriz-fen} apresenta a lógica de tradução da matriz interna da Godot para a notação FEN, essencial para que o Stockfish compreenda a posição das peças.

\begin{algorithm}[H]
	\caption{Lógica de Conversão da Matriz do Tabuleiro para Notação FEN.}
	\label{alg:matriz-fen}
	\Entrada{matriz\_do\_tabuleiro}
	\Inicio{
		texto\_fen $\leftarrow$ texto vazio\;
		\Para{cada linha da matriz\_do\_tabuleiro}{
			casas\_vazias $\leftarrow$ 0\;
			\Para{cada coluna da linha atual}{
				peca\_atual $\leftarrow$ verificar a peça na posição atual\;
				\Se{a casa estiver vazia}{
					casas\_vazias $\leftarrow$ casas\_vazias + 1\;
				}
				\Senao{
					\Se{casas\_vazias > 0}{
						concatenar o número de casas\_vazias ao texto\_fen\;
						casas\_vazias $\leftarrow$ 0\;
					}
					letra\_peca $\leftarrow$ converter a peca\_atual para sua letra correspondente\;
					concatenar letra\_peca ao texto\_fen\;
				}
			}
			\Se{casas\_vazias > 0}{
				concatenar o número de casas\_vazias ao texto\_fen\;
			}
			\Se{a linha atual não for a última}{
				concatenar o símbolo "/" ao texto\_fen\;
			}
		}
		\Retorna{texto\_fen}\;
	}
\end{algorithm}

O processo de decisão da IA é executado em uma \textit{Thread} separada para evitar o congelamento da interface. O Algoritmo \ref{alg:ponte_python} descreve a lógica de como o \textit{middleware} processa a requisição baseada no nível de dificuldade.

\begin{algorithm}[H]
	\caption{Lógica de Processamento de Movimento da Inteligência Artificial.}
	\label{alg:ponte_python}
	\Entrada{estado\_do\_tabuleiro, nivel\_dificuldade}
	\Inicio{
		ler o estado\_do\_tabuleiro\;
		\Se{nivel\_dificuldade == 1}{
			lista\_movimentos $\leftarrow$ obter todos os movimentos válidos\;
			movimento\_escolhido $\leftarrow$ sortear aleatoriamente um movimento da lista\_movimentos\;
		}
		\Senao{
			iniciar o motor de xadrez\;
			\Se{nivel\_dificuldade == 2}{
				limite\_busca $\leftarrow$ profundidade 1\;
			}
			\Senao{
				limite\_busca $\leftarrow$ profundidade 20\;
			}
			movimento\_escolhido $\leftarrow$ calcular a melhor jogada usando o limite\_busca\;
		}
		resposta $\leftarrow$ formatar movimento\_escolhido e status de sucesso\;
		\Retorna{resposta}\;
	}
\end{algorithm}
